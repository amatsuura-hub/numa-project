## 3. 2026-04-16 の大規模変更要点（before / after）

### 3-1. カスタムドメイン適用

| | Before | After |
|---|---|---|
| フロント URL | `https://d1vt9zgiodctts.cloudfront.net` | `https://numa-roadmap.com`（旧 URL も引き続き動作） |
| API URL | `https://1t384b0i24.execute-api.ap-northeast-1.amazonaws.com/dev` | `https://api.numa-roadmap.com`（旧 URL も引き続き動作） |

### 3-2. WAF 廃止

CloudFront WAF (`aws_wafv2_web_acl.cloudfront`) と API Gateway WAF (`aws_wafv2_web_acl.api` + association) を Terraform から削除し Phase 1 apply で実環境からも削除。関連変数 (`cloudfront_waf_rate_limit`, `waf_rate_limit`) も削除。動機: コスト削減。

### 3-3. Lambda 監視強化（Phase 1 apply で実環境反映）

- DLQ (SQS `dev-numa-api-dlq`, 14 日保持) 新規作成・Lambda から紐付け
- X-Ray Active トレーシング有効化
- CloudWatch アラーム 6 種 + SNS トピック `dev-numa-alarms` を新規作成
- Lambda CloudWatch Log Groups（従来は Lambda が自動作成、無期限保持）を Terraform 管理化 → 30 日保持に統一
  - 既存ロググループは `terraform import` で state に取り込み済み

### 3-4. リモートステート bootstrap

S3 バケット `numa-terraform-state` + DynamoDB lock table `numa-terraform-lock` を Phase 1 apply で新規作成。ただし `backend.tf` の `backend "s3"` ブロック有効化と `-migrate-state` は **未実施**。

### 3-5. API Gateway redeployment trigger バグ修正

`aws_api_gateway_deployment.triggers` の hash に `aws_api_gateway_method_response.proxy_options` と `aws_api_gateway_integration_response.proxy_options` を追加。これらを欠いていたため CORS origin 変更がステージに自動反映されず、今回の apply 後に `aws apigateway create-deployment` で手動再配備する事象が発生。以降は Terraform だけで反映される。

### 3-6. その他コード整理

  -H "Origin: https://numa-roadmap.com" \
  -H "Access-Control-Request-Method: GET" \
  https://api.numa-roadmap.com/api/health \
  | grep -iE "^(HTTP|access-control)"

# 配信 JS に埋め込まれた VITE_API_URL の確認
curl -s https://numa-roadmap.com/ | grep -oE "assets/index-[A-Za-z0-9_-]+\.js" | head -1
# → 返ったパスを curl して `https://api.numa-roadmap.com` が含まれることを確認

## 7. 引き継ぎ時の注意点

1. **単一環境運用**: dev / prod を Terraform 的には分離可能だが、現状は root `terraform.tfvars` + local state に統一。`environments/prod/` は未使用の雛形で、そのまま apply すると重複した Route 53 ゾーンを作ろうとして失敗する可能性あり。
2. **NS レコードはレジストラ側**: Terraform の管轄外。Route 53 ゾーンを再作成すると NS が変わるので、その場合はレジストラでの再登録が必要。
3. **API Gateway の stage 再デプロイ**: 今回のコミットで `triggers` に response 系を含める修正が入ったので、通常は Terraform のみで完結するが、もし CORS/response の変更が stage に反映されない場合は `aws apigateway create-deployment --rest-api-id 1t384b0i24 --stage-name dev` で強制配備する。
4. **CloudFront ログ配送**: 新 S3 logs バケットは `BucketOwnerPreferred` + `log-delivery-write` ACL で設定。CloudFront の legacy standard logging を使っているため、`BucketOwnerEnforced` には変更しないこと。
5. **deploy.yml の smoke test**: API が Cognito Authorizer で全エンドポイント保護しているため `/api/health` も 401 が返る。workflow は「何らかのレスポンスが返れば OK」として扱う実装（`HTTP_CODE != "000"`）。
